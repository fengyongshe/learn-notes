Apache Slider项目调研

转至元数据结尾
  • 应用页面限制
  • 附件：2
  • 被郭磊涛添加，被郭磊涛最后更新于十月 27, 2014  (查看更改)
转至元数据起始
Apache Slider 项目源自于Hoya项目，其目标是提供一个简单的方法使得HBase可以运行在YARN的环境中，即HBase on YARN。 Apache Slider 则将Hoya进行通用化，使用Slider可以将一些长运行应用(long-running apps，例如HBase/Storm等)进行打包、部署和管理，简化将Apps部署到YARN以及生命周期管理的编码工作。将HBase运行在YARN上将带来众多好处，包括：
  1. 在一个物理集群中可同时部署多个HBase集群实例
  2. 为HBase集群提供资源隔离（这一点HBase本身做不到）
  3. 将多个版本的HBase集群部署到一个物理集群中

截至2014年10月，Slider 还是Apache的incubator项目，正式release版本为 0.40-incubating，trunk版本是0.5.2-incubating版本。在0.5.2版本中，Slider已经内置了将Accumulo、HBase、Storm和memcached应用进行打包并可以直接部署到YARN环境中。

其他应用如果也采用Slider在YARN部署的话，需要采用如下的步骤：

  1. 首先将Apps按照Slider的规范打包，目前Slider-0.5.2版本已经将Accumulo、HBase、Storm和memcached打包好了。
  2. 将打包好的Apps压缩文件上传到HDFS上。
  3. 配置Slider使之可以访问HDFS、YARN和ZK。
  4. 通过 bin/slider create + param 来提交运行Apps。

如步骤1提到的Apps按照Slider规范进行打包的方法可以参考：http://slider.incubator.apache.org/docs/slider_specs/hello_world_slider_app.html，具体如下：

  1. 创建meteinfo.xml文件，内容包括：Apps的名称、版本、外部端口、包含的组件及实例数限制(min/max)、组件的启停顺序以及启停的python脚本、适用的OS 以及 Apps的压缩文件路径。详细规范可参考： http://slider.incubator.apache.org/docs/slider_specs/application_definition.html
  2. 创建Apps的压缩文件。把Apps发布的 .tar.gz 下载下来，删除掉src/ 源文件目录再打包就可以了。也可以重新编译成package。
  3. 创建resources.json文件，内容包括：该app中各个组件的资源需求，包括<componentsName, memory, vcores, role_priority, #instances>。还可以配置app在某段时间内失败N次的错误检测策略。详细规范可以参考： http://slider.incubator.apache.org/docs/slider_specs/resource_specification.html
  4. 创建appConfig.json文件，以下字段是必须要定义的：application.def：Apps打包压缩文件在HDFS上的路径；java_home：JDK路径；package_list：Apps原始文件的路径；config_types：配置文件名称；site.global.app_user：运行应用的user；site.global.app_root：Apps压缩文件解压后的路径。另外还可以定义 site.global.* 的应用相关的配置参数。详细规范可以参考： http://slider.incubator.apache.org/docs/slider_specs/application_instance_configuration.html
  5. 实现metainfo.xml定义的Apps相关各组件操作的脚本。详细规范可以参考： http://slider.incubator.apache.org/docs/slider_specs/writing_app_command_scripts

利用Slider将HBase运行在YARN上的方法：http://slider.incubator.apache.org/docs/getting_started.html

Apache Slider主要包括3个组件： SliderClient、SliderAppMaster和Apps。

  1. SliderClient：与SliderAppMaster和YARN通过REST/RPC通信，对外提供CLI和底层API。

  2. SliderAppMaster：根据App配置向YARN申请container，每个component对应一个container，并监控container的状态。

  3. Apps：需要提供resource.json (各个components以及mem/cpu资源需求)、appsConfig (各个components的配置信息)、.zip (app应用压缩文件)。

其中SliderAppMaster又由如下几个模块组成：

  • AM Engine： 与SliderClient和YARN的接口。与RM交互请求和释放container，与NM交互启停container。当集群状态变化时，AM Engine将该信息发送给Apps State，后者返回相应的action，例如请求释放container或请求在某个特定节点上的资源。之后，AM Engine调用 provider来创建Apps启动环境并启动。

  • Provider：用于App的部署和启动。provider会在container中首先部署一个agent，该agent调用provider的REST API来执行返回的命令。具体包括，从HDFS上下载并解压app压缩文档，运行python脚本来启动服务。

  • Apps State：应用实例的信息，包括：resource需求说明、当前各role运行在哪些实例 instance的对应关系信息、 Role History用以记录哪些role部署在哪些nodes上的信息，该信息持久化存储到磁盘中，以便后续快速启动服务。

分析：

apache 可以简化Apps在YARN上的部署和启停，只需要撰写Apps各项服务的描述以及启停脚本即可。但是因为目前项目还处于incubator状态，而且经过测试hbase 经常出现“Unstable Application Instance : - failed with role HBASE_MASTER failing 3 times”的错误，所以系统还不太稳定。可以等待系统正式成为top project后再考虑部署。
